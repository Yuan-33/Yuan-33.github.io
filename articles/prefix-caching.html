    Prefix Caching

    KV Cache — Prefix Reuse & RadixAttention
    Prefix Caching 是一种跨请求复用 KV Cache 的优化技术。当多个请求共享相同的前缀（如系统提示词、Few-shot 示例）时，只需计算一次前缀的 KV Cache，后续请求直接复用，避免重复的 Prefill 计算。SGLang 的 RadixAttention 将这一思想推广为基于 Radix Tree 的通用前缀共享框架。

    核心原理

    在 LLM 服务中，大量请求共享相同的前缀——例如同一个应用的所有请求都使用相同的系统提示词（System Prompt），或者同一个 RAG 管道中的多个请求使用相同的检索上下文。传统推理系统对每个请求独立执行 Prefill，即使前缀完全相同也要重复计算。Prefix Caching 的核心思想是：将前缀的 KV Cache 缓存起来，当新请求的前缀与缓存匹配时，直接跳过 Prefill 阶段，从缓存位置开始生成。

    Prefix 匹配策略

    Prefix Caching 的关键挑战是如何高效地匹配前缀。最简单的方案是精确匹配——对前缀的 token 序列计算哈希值，在缓存中查找。但这种方式只能匹配完全相同的前缀，无法处理「前缀 A 是前缀 B 的子串」的情况。更先进的方案是使用 Radix Tree（基数树）来组织缓存：每个节点代表一个 token 序列片段，从根到叶的路径构成一个完整的前缀。新请求到达时，从 Radix Tree 的根开始逐 token 匹配，找到最长的公共前缀，直接复用其 KV Cache。SGLang 的 RadixAttention 就采用了这种方案。

    RadixAttention 详解

    SGLang 的 RadixAttention 将 Prefix Caching 推广为一个通用的前缀共享框架。核心数据结构是一棵 Radix Tree，每个节点存储一段 token 序列及其对应的 KV Cache 物理块。Radix Tree 支持三种操作：（1）查找——从根开始匹配最长公共前缀，返回匹配长度和对应的 KV Cache；（2）插入——将新的前缀及其 KV Cache 插入树中；（3）淘汰——当缓存空间不足时，使用 LRU 策略淘汰最近最少使用的节点。RadixAttention 还支持多级前缀共享——例如系统提示词是一级前缀，RAG 检索结果是二级前缀，用户输入是三级前缀，不同级别的前缀可以独立缓存和复用。

    Prefix Caching 的收益分析

    Prefix Caching 的收益取决于前缀的共享程度和长度。在以下场景中收益最大：（1）系统提示词共享——同一个应用的所有请求共享相同的系统提示词（通常 500-2000 token），Prefix Caching 可以将 Prefill 时间减少 50-80%；（2）Few-shot 示例共享——所有请求使用相同的 Few-shot 示例（通常 1000-3000 token），Prefill 时间减少 60-90%；（3）多轮对话——同一会话的后续轮次可以复用前几轮的 KV Cache，Prefill 只需处理新增的用户输入；（4）Agent 调用——同一个 Agent 的多次工具调用共享相同的上下文，Prefill 开销大幅降低。

    与 PagedAttention 的协同

    Prefix Caching 与 PagedAttention 天然兼容。在 vLLM 中，Prefix Caching 通过在 PagedAttention 的块表层面实现：共享前缀的多个请求的块表指向相同的物理块（通过引用计数管理），只有不共享的部分才分配新的物理块。这与 Copy-on-Write 的机制完全一致——共享时只读，需要修改时才复制。SGLang 的 RadixAttention 也基于类似的设计，Radix Tree 的每个节点对应 PagedAttention 的一组物理块。

    关键特性

    • 前缀复用：共享相同前缀的多个请求复用同一份 KV Cache，避免重复 Prefill 计算，Prefill 时间减少 50-90%
    • Radix Tree 组织：使用 Radix Tree 管理缓存，支持最长前缀匹配和多级前缀共享
    • TTFT 大幅降低：在系统提示词共享场景下，首 Token 延迟（TTFT）降低 50-80%
    • 多级缓存：支持系统提示词、RAG 上下文、用户输入等多级前缀独立缓存和复用
    • 与 PagedAttention 兼容：在块表层面实现前缀共享，通过引用计数管理物理块的生命周期
    • LRU 淘汰策略：缓存空间不足时自动淘汰最近最少使用的前缀，保证缓存命中率

    面试常问 Q&A

    Q1：Prefix Caching 和 KV Cache 有什么区别？

    A：KV Cache 是单个请求内部的优化——缓存当前请求已经计算过的 Key/Value，避免在自回归生成中重复计算。Prefix Caching 是跨请求的优化——缓存不同请求之间共享的前缀 KV Cache，避免多个请求重复计算相同的前缀。两者是不同层面的优化，可以同时使用。

    Q2：RadixAttention 的 Radix Tree 是如何工作的？

    A：Radix Tree 是一种压缩前缀树，每个节点代表一段 token 序列（而非单个 token）。新请求到达时，从根节点开始逐段匹配，找到最长的公共前缀节点，直接复用其 KV Cache。匹配结束后，将不匹配的部分作为新节点插入树中。当缓存空间不足时，使用 LRU 策略淘汰最近最少使用的叶子节点。Radix Tree 的查找复杂度为 O(前缀长度/段长度)，非常高效。

    Q3：Prefix Caching 在多轮对话中如何工作？

    A：在多轮对话中，每一轮的输入包含之前所有轮次的历史。Prefix Caching 可以缓存前 N-1 轮的 KV Cache，当第 N 轮到达时，只需 Prefill 第 N 轮新增的用户输入，历史部分直接从缓存中加载。这使得多轮对话的 Prefill 时间与轮次无关，始终保持在单轮的水平。实现上，每一轮结束后将其 KV Cache 注册到 Radix Tree 中，下一轮到达时自动匹配。

    Q4：Prefix Caching 的缓存一致性如何保证？

    A：Prefix Caching 的缓存是只读的——一旦前缀的 KV Cache 被缓存，就不会被修改。共享同一前缀的多个请求通过引用计数管理物理块：当所有引用都释放时，物理块才被回收。这避免了缓存一致性问题。在 vLLM 中，Prefix Caching 与 PagedAttention 的 Copy-on-Write 机制结合，共享块只读，需要修改时（如序列分叉）才复制。

    Q5：什么场景下 Prefix Caching 收益最大？

    A：收益最大的场景是：（1）大量请求共享长系统提示词——如企业级 LLM 应用，所有请求使用相同的 1000+ token 系统提示词；（2）RAG 应用——多个请求使用相同的检索上下文；（3）多轮对话——同一会话的后续轮次复用历史 KV Cache；（4）Agent 调用——同一个 Agent 的多次工具调用共享上下文。在这些场景下，Prefix Caching 可以将 Prefill 时间减少 50-90%，显著降低 TTFT。
