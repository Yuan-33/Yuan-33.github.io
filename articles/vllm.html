    vLLM

    Inference Engine — High-Throughput LLM Serving
    vLLM 是由 UC Berkeley 开源的高性能 LLM 推理引擎，核心创新是 PagedAttention——借鉴 OS 虚拟内存分页机制管理 KV Cache，将显存利用率从 20-40% 提升至接近 100%。vLLM 集成了 Continuous Batching、Prefix Caching、Chunked Prefill、多种并行策略等现代推理优化技术，是目前最广泛使用的开源 LLM 推理框架。

    架构概述

    vLLM 的架构分为三层：（1）调度层（Scheduler）——负责请求的接收、排队、调度决策，实现 Continuous Batching 和抢占；（2）引擎层（LLMEngine）——管理模型的前向传播、KV Cache 分配、采样策略；（3）Worker 层——在每个 GPU 上执行实际的计算，支持 TP 和 PP 并行。vLLM 使用 OpenAI 兼容的 API 服务器，可以作为 LLM 服务的 drop-in 替代。

    核心特性：PagedAttention

    PagedAttention 是 vLLM 的核心创新。传统推理系统为每个请求预分配最大序列长度的连续显存，导致大量内存碎片和浪费。PagedAttention 将 KV Cache 划分为固定大小的物理块（Block，默认 16 token/块），通过块表（Block Table）维护逻辑块到物理块的映射，按需分配和释放显存。这将显存利用率从 20-40% 提升至接近 100%，相同显存下可服务的并发请求数量提升 2-4 倍。PagedAttention 还支持 Copy-on-Write——多个采样序列共享前缀 KV Cache，分叉时才复制，在 best-of-N 场景下节省大量显存。

    Continuous Batching 实现

    vLLM 实现了迭代级调度（Iteration-Level Scheduling），在每个 Decode 步骤动态调度请求：完成的请求立即释放资源，新请求随时加入批次。调度器维护两个队列：waiting 队列（等待 Prefill 的新请求）和 running 队列（正在 Decode 的请求）。每轮调度时，调度器根据 GPU 显存预算决定从 waiting 队列中取出多少请求执行 Prefill，同时对 running 队列中的所有请求执行 Decode。当显存不足时，调度器可以抢占（Preempt）低优先级请求——将其 KV Cache Swap 到 CPU 内存或直接丢弃（Recompute）。

    Prefix Caching 与 Chunked Prefill

    vLLM v0.4+ 支持 Automatic Prefix Caching（APC）——自动缓存请求的前缀 KV Cache，当新请求的前缀与缓存匹配时跳过 Prefill。APC 基于 PagedAttention 的块表实现——共享前缀的请求的块表指向相同的物理块（引用计数管理）。vLLM v0.5+ 引入 Chunked Prefill 支持——将长 Prompt 的 Prefill 拆分为多个小块与 Decode 交错执行，消除 Prefill Stall，降低 Decode 请求的 TTBT。

    并行策略支持

    vLLM 支持多种并行策略：（1）Tensor Parallelism（TP）——层内切分，通过 NCCL AllReduce 通信，适用于节点内多 GPU；（2）Pipeline Parallelism（PP）——层间切分，适用于跨节点场景；（3）Data Parallelism（DP）——多个 vLLM 实例独立运行，通过负载均衡分发请求。vLLM 的 TP 实现基于 Megatron-LM 的列并行+行并行方案，支持 NVLink 和 PCIe 两种互联模式。

    支持的模型与硬件

    vLLM 支持主流的 LLM 架构：LLaMA/LLaMA-2/3、Mistral、Qwen、ChatGLM、Falcon、GPT-NeoX、BLOOM 等。硬件方面支持 NVIDIA GPU（A100/H100/H200/L40S）、AMD GPU（MI250/MI300）、AWS Inferentia、Google TPU。量化方面支持 GPTQ、AWQ、SqueezeLLM、FP8 等多种量化格式。vLLM 还支持 LoRA 适配器的热加载——无需重启服务即可切换不同的 LoRA 模型。

    关键特性

    • PagedAttention：分页显存管理，显存利用率接近 100%，并发请求数量提升 2-4x
    • Continuous Batching：迭代级调度，每步动态添加/移除请求，GPU 利用率 80-95%
    • Automatic Prefix Caching：自动缓存前缀 KV Cache，多轮对话/RAG 场景 TTFT 大幅降低
    • Chunked Prefill：长 Prompt 拆分执行，消除 Prefill Stall，TTBT 降低 3-5x
    • TP/PP/DP 并行：支持多种并行策略，单节点 8 GPU 到多节点集群均可部署
    • OpenAI 兼容 API：提供 drop-in 替代的 API 服务器，支持 Chat Completions、Completions、Embeddings
    • LoRA 热加载：支持多个 LoRA 适配器的动态切换，无需重启服务
    • 多种量化：支持 GPTQ、AWQ、FP8、SqueezeLLM 等量化格式

    面试常问 Q&A

    Q1：vLLM 的核心优势是什么？为什么比 HuggingFace Transformers 快？

    A：核心优势是 PagedAttention + Continuous Batching。HuggingFace Transformers 使用静态 batching（等待整个批次完成）和连续显存预分配（大量碎片），GPU 利用率仅 20-40%。vLLM 通过分页显存管理消除碎片（利用率接近 100%），通过迭代级调度消除木桶效应（GPU 利用率 80-95%）。在高并发场景下，vLLM 的吞吐量是 HuggingFace 的 5-24 倍。

    Q2：vLLM 的 PagedAttention 和操作系统的虚拟内存有什么相似之处？

    A：完全类似。OS 虚拟内存将进程的地址空间划分为固定大小的页（Page），通过页表映射到物理内存的帧（Frame），支持非连续存储和按需分配。PagedAttention 将请求的 KV Cache 划分为固定大小的块（Block），通过块表映射到 GPU 显存的物理块，同样支持非连续存储和按需分配。两者都消除了连续分配导致的碎片问题。Copy-on-Write 机制也是直接借鉴 OS 的 COW。

    Q3：vLLM 如何处理显存不足的情况？

    A：vLLM 的调度器会进行抢占（Preemption）。当 GPU 显存不足以容纳新请求的 KV Cache 时，调度器会暂停（Swap）或丢弃（Recompute）低优先级的请求。Swap 将被抢占请求的 KV Cache 从 GPU 交换到 CPU 内存，恢复时再加载回来。Recompute 直接丢弃 KV Cache，恢复时重新执行 Prefill。调度器优先 Swap（恢复快），当 CPU 内存也不足时使用 Recompute。

    Q4：vLLM 和 TensorRT-LLM 的区别是什么？

    A：vLLM 是 Python 原生实现，易用性好，支持模型广泛，开源社区活跃。TensorRT-LLM 是 NVIDIA 的推理引擎，基于 TensorRT 深度优化，kernel 级别优化更极致（如 FP8/INT4 量化、自定义 Attention kernel），在 NVIDIA GPU 上性能通常更高。但 TensorRT-LLM 的模型支持范围较窄（需要逐模型适配），部署复杂度更高。选择建议：追求易用性和模型兼容性选 vLLM，追求极致性能且只用 NVIDIA GPU 选 TensorRT-LLM。

    Q5：vLLM 的 Prefix Caching 是如何工作的？

    A：vLLM 的 Automatic Prefix Caching（APC）基于 PagedAttention 的块表实现。对于每个请求的 KV Cache 物理块，计算其内容的哈希值。当新请求到达时，逐块匹配哈希值，找到最长的公共前缀。匹配到的物理块直接复用（通过引用计数管理），未匹配的部分分配新物理块进行 Prefill。APC 在多轮对话（复用历史 KV Cache）和系统提示词共享（所有请求复用同一系统提示词）场景下效果显著。
