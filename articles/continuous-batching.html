    Continuous Batching

    Inference Framework — Iteration-Level Scheduling
    Continuous Batching（也称 Iteration-Level Scheduling）是 LLM 推理服务的核心调度策略。与传统 Static Batching 需要等待整个批次完成不同，Continuous Batching 在每个 Decode 步骤动态调度请求——已完成的请求立即释放资源，新请求随时加入批次，最大化 GPU 利用率和系统吞吐量。这一技术由 Orca（OSDI 2022）首次提出，现已成为 vLLM、TensorRT-LLM、SGLang 等所有主流推理框架的标配。

    核心原理

    LLM 推理分为两个阶段：Prefill（预填充）处理输入 prompt 生成 KV Cache，是 compute-bound；Decode（解码）逐 token 生成输出，是 memory-bound。传统 Static Batching 将多个请求组成一个批次，等待所有请求都生成完毕后才释放资源。这导致严重的资源浪费——如果批次中有一个请求需要生成 1000 个 token，其他请求即使只生成 10 个 token 也必须等待。

    Static Batching 的问题

    Static Batching 的核心问题是「木桶效应」——整个批次的速度取决于最长的序列。具体表现为：（1）GPU 利用率低——当批次中部分请求已完成、部分仍在生成时，已完成请求的 GPU 资源（显存、计算单元）处于空闲状态，但无法被新请求使用；（2）延迟不可控——新请求必须等待当前批次全部完成才能被处理，队列延迟不可预测；（3）显存浪费——所有请求都预分配最大序列长度的 KV Cache，即使实际生成长度远小于最大长度。

    Continuous Batching 的调度机制

    Continuous Batching 在每个 Decode 步骤（iteration）进行调度决策：（1）检查当前批次中是否有请求已完成生成（遇到 EOS token 或达到最大长度），已完成的请求立即从批次中移除，释放其 KV Cache 和 GPU 资源；（2）从等待队列中取出新请求，执行 Prefill 后加入当前批次；（3）对当前批次中的所有请求执行一次 Decode 步骤，每个请求生成一个 token；（4）重复上述过程。这种「每步调度」的方式使得 GPU 始终保持高利用率——没有请求等待，没有资源空闲。

    Prefill 与 Decode 的混批

    Continuous Batching 的一个关键设计决策是如何处理 Prefill 和 Decode 的混合。由于 Prefill 是 compute-bound（大量矩阵乘法），Decode 是 memory-bound（大量显存读取），两者的计算特性差异很大。早期的实现（如 Orca）将 Prefill 和 Decode 放在同一个批次中执行，但这会导致 Prefill 的长计算阻塞 Decode 的低延迟需求。现代框架（如 vLLM、SGLang）采用「Prefill-Decode 分离」策略：在调度时，新请求的 Prefill 和已有请求的 Decode 交替执行，或者将 Prefill 拆分为多个小块（Chunked Prefill）与 Decode 交错执行，避免 Prefill 阻塞 Decode。

    调度策略优化

    Continuous Batching 的调度策略直接影响系统的吞吐量和延迟。常见的优化策略包括：（1）FIFO（先来先服务）——最简单，但可能导致短请求被长请求阻塞；（2）Shortest Job First（SJF）——优先处理预估生成长度短的请求，降低平均延迟；（3）Preemptive Scheduling（抢占式调度）——当 GPU 资源紧张时，暂停低优先级请求，优先处理高优先级请求，暂停的请求稍后恢复（通过 Swap 或重新 Prefill）；（4）Fair Scheduling（公平调度）——保证每个请求获得公平的 GPU 时间，避免饥饿。

    与 PagedAttention 的协同

    Continuous Batching 的高效运行依赖 PagedAttention 的内存管理。当请求被移除时，PagedAttention 将其物理块归还空闲池；当新请求加入时，从空闲池按需分配物理块。没有 PagedAttention，Continuous Batching 的内存管理会非常复杂（需要处理连续显存的分配和回收）。两者结合使得 vLLM 等框架可以同时服务数百个并发请求，GPU 利用率接近 100%。

    关键特性

    • 每步调度：在每个 Decode 步骤动态添加/移除请求，消除 Static Batching 的木桶效应
    • GPU 利用率接近 100%：已完成请求立即释放资源，新请求随时加入，GPU 始终保持忙碌
    • 吞吐量提升 2-10x：相比 Static Batching，吞吐量提升 2-10 倍（取决于请求长度方差）
    • 更低的延迟：新请求无需等待当前批次完成，队列延迟大幅降低
    • 支持抢占：资源紧张时可以暂停低优先级请求，支持 Swap 到 CPU 或重新 Prefill
    • 行业标配：vLLM、TensorRT-LLM、SGLang、LMDeploy、DeepSpeed-FastGen 等所有主流框架均已支持

    面试常问 Q&A

    Q1：Continuous Batching 和 Static Batching 的核心区别是什么？

    A：Static Batching 在批次级别调度——将多个请求组成一个批次，等所有请求完成后才释放资源、处理下一批。Continuous Batching 在迭代级别调度——每个 Decode 步骤都进行调度决策，完成的请求立即移除，新请求随时加入。核心区别是调度粒度：Static 是「批次级」，Continuous 是「步级」。这消除了木桶效应，使得 GPU 利用率从 30-60% 提升至 80-95%。

    Q2：Continuous Batching 中 Prefill 和 Decode 如何协调？

    A：Prefill 是 compute-bound，Decode 是 memory-bound，混批会导致 Prefill 阻塞 Decode。现代框架有两种处理方式：（1）交替执行——在调度层面，先执行一轮 Prefill（处理新请求），再执行一轮 Decode（处理已有请求），交替进行；（2）Chunked Prefill——将长 Prefill 拆分为多个小块，与 Decode 交错执行在同一轮中。后者更细粒度，可以更好地平衡 Prefill 和 Decode 的延迟。

    Q3：Continuous Batching 如何处理请求的抢占（Preemption）？

    A：当 GPU 资源（主要是显存）不足以容纳所有活跃请求时，需要抢占部分请求。抢占的方式有两种：（1）Swap——将被抢占请求的 KV Cache 从 GPU 交换到 CPU 内存，释放 GPU 显存给其他请求，稍后恢复时再从 CPU 加载回来；（2）Recompute——直接丢弃被抢占请求的 KV Cache，恢复时重新执行 Prefill 重建 KV Cache。Swap 的恢复更快但需要 CPU 内存，Recompute 不需要额外内存但恢复时间更长。

    Q4：为什么 Continuous Batching 需要 PagedAttention 的配合？

    A：Continuous Batching 的动态调度意味着请求的添加和移除是频繁且不可预测的。如果使用传统的连续显存分配方案，频繁的分配和回收会导致严重的内存碎片。PagedAttention 的分页机制天然适合这种动态场景——物理块可以被任意请求使用，添加请求时分配块，移除请求时释放块，没有碎片问题。两者结合是现代 LLM 推理系统的基础架构。

    Q5：Continuous Batching 的吞吐量提升来自哪里？

    A：主要来自三个方面：（1）消除木桶效应——不再等待最长序列完成，GPU 利用率大幅提升；（2）资源即时回收——完成的请求立即释放显存，新请求可以立即使用，没有空闲期；（3）批处理效率——GPU 的并行计算能力在 batch size 较大时更高效，Continuous Batching 通过动态调度维持较大的有效 batch size。实测在真实负载下，吞吐量提升 2-10 倍。
