Eagle-v2

Speculative Decoding — Dynamic Draft Tree
Eagle-v2（EAGLE-2: Faster Inference of Language Models with Dynamic Draft Trees）在 Eagle-v1 的基础上，核心发现是 Draft token 的接受率不仅与位置相关，还与上下文高度相关。利用 EAGLE Draft 模型的良好校准性（置信度分数能以小误差近似真实接受率），Eagle-v2 引入了上下文感知的动态 Draft Tree，将计算资源集中在高概率路径上，实现了 3.05x-4.26x 的端到端无损加速，比 Eagle-v1 快 20%-40%。

核心改进

Eagle-v1 使用静态 Draft Tree，隐式假设 Draft token 的接受率仅取决于其位置。Eagle-v2 通过实验推翻了这一假设：接受率还与上下文强相关——不同上下文下，同一位置的 Draft token 接受率差异很大。基于这一洞察，Eagle-v2 利用 EAGLE Draft 模型的良好校准性（置信度分数与真实接受率误差很小），提出上下文感知的动态 Draft Tree，根据每个节点的置信度自适应调整树结构，实现更高效的计算分配。

从静态树到动态树的演进

Eagle-v1 的静态树结构存在明显的计算浪费：所有分支的宽度和深度固定，低置信度分支消耗了大量无效计算。Eagle-v2 的核心思想是「资源向高概率路径倾斜」——根据 Draft Head 输出的 token 概率分布，动态决定每个节点的子节点数量（宽度）和是否继续扩展（深度）。具体来说，Eagle-v2 将候选 token 按概率降序排列，设置概率阈值 θ：概率 ≥ θ 的 token 扩展子节点，概率 < θ 的作为叶节点终止。同时，每个节点的子节点数量有上限（通常 2-4 个），总节点数受预算约束（通常 20-60 个），在预算内最大化期望接受 token 数量。

Tree Attention 验证机制详解

Tree Attention 是 Eagle-v2 验证阶段的核心优化，支持将整个 Draft Tree 作为一个批次进行并行验证。其关键设计包括：

树的线性化：将树结构展平为一个序列，每个节点维护一个 mask 向量，标识它可以 attend 到哪些祖先节点——根节点 attend 到所有前缀 token，子节点只 attend 到其路径上的祖先节点和前缀 token。

位置编码处理：同一深度的节点共享相同的相对位置，不同深度的节点位置递增。这保证了注意力计算的正确性，同时避免了位置编码冲突。

验证策略：主模型对整个树进行一次前向传播，得到每个节点的输出分布。然后从根到叶遍历，对每个节点比较主模型分布和 Draft 分布：若接受则继续向下验证子节点；若拒绝则从该节点的修正分布中采样，该分支以下的所有子节点全部丢弃。最终得到一条从根到某叶节点的接受路径，路径上的所有 token 即为本轮生成的结果。

动态树的构建算法

Eagle-v2 的动态树构建是一个贪心迭代过程，核心是从根节点出发，每一步都全局筛选概率最高的候选分支、淘汰低置信度路径，最终生成一棵非对称、高命中的候选token树，彻底解决了Eagle-v1固定树结构的算力浪费问题。

图：Eagle-v2 动态树构建、树掩码生成与投机解码全流程示意图

  1. 根节点初始化：以Target大模型最后一步输出的采样token作为树的根节点，同时传入Target模型输出的最后一层隐状态，作为Draft模型的启动输入。
  2. 首层候选生成：Draft模型基于根节点隐状态完成前向推理，通过Draft Head输出token概率分布，取Top-K个概率最高的token作为树的第一层叶子节点，同步记录每个节点的累计对数概率、父节点索引。
  3. 全局贪心分支筛选：对当前层所有候选节点的累计概率做全局排序，仅保留Top-K个置信度最高的分支，直接淘汰其余低概率路径，避免算力浪费在低命中的候选上。
  4. 层级迭代扩展：以筛选后的Top-K分支作为新的父节点，重复「前向推理→概率输出→Top-K采样→全局排序筛选」的流程，迭代Depth次，最终生成一棵深度为Depth、总候选token数固定的动态树。
  5. 树结构元数据生成：树构建完成后，同步生成3个核心元数据，支撑Target模型的单步并行验证：
    • tree_mask：树的因果注意力掩码，确保每个节点仅能关注自身的祖先路径，杜绝跨分支信息泄露
    • tree_position_ids：每个节点在树中的相对位置编码，完美适配RoPE位置编码规则
    • retrieve_indices：树中所有完整「根→叶子」路径的索引，用于Target模型验证后的最优路径选择
  6. 批量并行验证：将整棵树的所有候选token一次性输入Target模型，通过树注意力机制完成单步并行推理，一次性验证所有候选token的概率，无需逐token自回归，最大化利用大模型的并行算力。

这套动态树构建算法，相比Eagle-v1的固定二叉树，在相同的Draft算力开销下，候选token的平均置信度显著提升，同时大幅降低了低概率分支的无效算力开销，是Eagle-v2实现更高Block Efficiency、更快端到端推理速度的核心设计。

与 Eagle-v1 的性能对比

论文在三个 LLM 系列（Vicuna、LLaMA2-Chat、Mixtral）、六个任务（对话、代码生成、数学推理等）上的评估显示，Eagle-v2 实现了 3.05x-4.26x 的加速比，比 Eagle-v1 快 20%-40%。具体数据：在 Vicuna-13B 上，Eagle-v1 的加速比约 3.0x，Eagle-v2 提升至 3.5x-4.0x；在 LLaMA2-Chat 70B 上，从 2.7x-3.5x 提升至 3.5x-4.26x。

关键特性

  • 上下文感知动态树构建：根据 Draft token 概率动态调整树的形状，高概率路径更深更宽，低概率路径提前终止，最大化有效 Draft 数量
  • Tree Attention 并行验证：将整个 Draft Tree 组织为一个批次，通过一次前向传播验证所有分支，充分利用 GPU 并行能力，验证开销与静态树相当
  • 接受率提升 20-40%：相比 Eagle-v1 的静态树结构，动态树在各类任务上平均接受率提升 20-40%，尤其在长文本生成场景下优势更大
  • 可配置树大小：支持设置最大树节点数（通常 20-60），在延迟与吞吐之间灵活权衡，适应不同部署场景
  • 端到端加速 3.05x-4.26x：在三个 LLM 系列、六个任务上实现 3.05x-4.26x 无损加速，比 Eagle-v1 快 20%-40%
  • 完全兼容 Eagle-v1：Draft Head 架构与 Eagle-v1 完全相同，仅修改了推理阶段的树构建策略，已有 Eagle-v1 的预训练权重可直接升级使用

面试常问 Q&A

Q1：Eagle-v2 的动态树和 Eagle-v1 的静态树有什么核心区别？

A：核心区别在于树结构的构建逻辑。Eagle-v1 使用静态树，隐式假设 Draft token 的接受率仅与位置相关，所有分支的宽度和深度预先固定，不考虑上下文差异。Eagle-v2 发现接受率还与上下文强相关，利用 Draft 模型的良好校准性（置信度分数能近似真实接受率），根据每个节点的实际置信度动态调整树结构：高概率节点获得更多子节点和更深的扩展，低概率节点提前终止。这使得在相同的节点预算下，Eagle-v2 的期望接受 token 数量显著多于静态树。

Q2：Tree Attention 的 mask 是如何设计的？

A：Tree Attention 的核心是因果 mask 的设计。将树线性化后，每个节点 u 有一个 mask 向量：对于树中的节点 u 和 v，mask[u][v] = 1 当且仅当 v 是 u 的祖先节点，或 v 位于 u 的左侧同一层级。这保证了每个节点只能看到其路径上的祖先和已经验证过的同层节点，保持了因果性。实际实现中，mask 矩阵是稀疏的，可以通过 sparse attention 优化计算效率。

Q3：动态树的概率阈值 θ 如何确定？

A：θ 的选择需要在「树的覆盖率」和「树的深度」之间权衡。θ 过高会导致树太窄（只有很少的分支），错过一些可能被接受的 token；θ 过低会导致树太宽太浅（很多低概率分支），浪费计算资源。论文中通过在验证数据集上进行网格搜索来确定最优 θ，通常在 0.1-0.3 之间。实践中也可以根据节点预算动态调整 θ：先用一个初始 θ 构建树，如果节点数未达预算则降低 θ 继续扩展。

Q4：Eagle-v2 的验证开销是否会因为树变大而增加？

A：不会。Eagle-v2 设定了固定的节点预算（例如 50 个节点），与 Eagle-v1 的静态树节点数相同。因此验证阶段的计算量不变，但由于动态树的接受率更高，每个验证轮次的有效输出 token 更多，所以端到端加速比更大。

Q5：Eagle-v2 相比 Eagle-v1 的训练成本有变化吗？

A：没有。Eagle-v2 的 Draft Head 架构与 Eagle-v1 完全相同，仅修改了推理阶段的树构建策略（从静态树改为动态树）。因此不需要重新训练 Draft Head，已有的 Eagle-v1 预训练权重可以直接用于 Eagle-v2。这是一个重要的工程优势——只需修改推理代码，无需额外训练成本。
