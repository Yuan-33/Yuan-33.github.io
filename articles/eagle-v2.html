Eagle-v2

Speculative Decoding — Dynamic Draft Tree
Eagle-v2 在 Eagle-v1 的基础上引入了动态 Draft Tree 结构，根据每个 Draft token 的置信度分数自适应调整树的宽度和深度，将计算资源集中在高概率路径上。这一改进使 Eagle-v2 的接受率和端到端加速比进一步提升，成为 2024 年投机解码的 SOTA 方案。

核心改进

Eagle-v1 使用固定的树结构进行 Draft，所有分支的宽度和深度相同。Eagle-v2 发现不同位置的 Draft token 置信度差异很大——高置信度 token 更可能被接受，应该扩展更深；低置信度 token 应该被剪枝。Eagle-v2 根据 Draft Head 输出的概率动态构建树，实现了更高效的计算分配。

从固定树到动态树的演进

Eagle-v1 的固定树结构存在明显的计算浪费：假设树宽为 w、深度为 d，则总节点数为 w^d，但实际被主模型接受的路径通常只有一条。大量低概率分支的 Draft 和验证都是无效计算。Eagle-v2 的核心思想是「把计算花在刀刃上」——根据 Draft Head 输出的 token 概率分布，动态决定每个节点应该扩展多少个子节点（宽度）以及是否继续向下扩展（深度）。具体地，Eagle-v2 将所有候选 token 按概率降序排列，设置一个概率阈值 θ：概率 ≥ θ 的 token 获得子节点继续扩展，概率 < θ 的 token 作为叶节点终止。同时，每个节点的子节点数量也有上限约束（通常 2-4 个），以控制总节点数不超过预算。

Tree Attention 验证机制详解

Tree Attention 是 Eagle-v2 验证阶段的核心优化。传统的投机解码每次只验证一条 Draft 序列，而树结构的 Draft 包含多条分支路径。Tree Attention 将整个 Draft Tree 组织为一个批次进行验证：

树的线性化：将树结构展平为一个序列，每个节点维护一个 mask 向量，标识它可以 attend 到哪些祖先节点。根节点 attend 到所有前缀 token，子节点只 attend 到其路径上的祖先节点和前缀 token。

位置编码处理：同一深度的节点共享相同的相对位置，不同深度的节点位置递增。这保证了注意力计算的正确性，同时避免了位置编码冲突。

验证策略：主模型对整个树进行一次前向传播，得到每个节点的输出分布。然后从根到叶遍历，对每个节点比较主模型分布和 Draft 分布：若接受则继续向下验证子节点；若拒绝则从该节点的修正分布中采样，该分支以下的所有子节点全部丢弃。最终得到一条从根到某叶节点的接受路径，路径上的所有 token 即为本轮生成的结果。

动态树的构建算法

Eagle-v2 的动态树构建是一个贪心过程：（1）从根节点开始，Draft Head 生成第一层候选 token 及其概率；（2）按概率排序，选择 Top-k 个候选作为第一层子节点；（3）对每个子节点，Draft Head 继续自回归生成下一层候选；（4）根据概率阈值 θ 决定是否扩展——高概率节点继续扩展，低概率节点终止；（5）重复直到达到最大深度或节点预算。树的总节点数受预算约束（通常 20-60 个节点），在预算内最大化期望接受 token 数量。论文中还引入了「置信度传播」机制——父节点的置信度会影响子节点的扩展决策，高置信度路径获得更多的计算资源。

与 Eagle-v1 的性能对比

在相同计算预算下（例如最大树节点数 50），Eagle-v2 的有效 Draft token 数量比 Eagle-v1 多 20-40%。具体数据：在 Vicuna-7B 上，Eagle-v1 的平均接受长度约 2.8 token/轮，Eagle-v2 提升到 3.5 token/轮；在 Vicuna-13B 上，从 3.1 提升到 3.9；在 Vicuna-70B 上，从 3.4 提升到 4.2。端到端加速方面，Eagle-v2 在 7B 模型上实现 3.0x，在 13B 上实现 3.5x，在 70B 上实现 4.0x 加速。

关键特性

  • 动态树构建：根据 Draft token 概率动态调整树的形状，高概率路径更深更宽，低概率路径提前终止，最大化有效 Draft 数量
  • Tree Attention 验证：将整个 Draft Tree 组织为一个批次，通过一次前向传播验证所有分支，充分利用 GPU 并行能力，验证开销与固定树相同
  • 接受率提升 20-40%：相比 Eagle-v1 的固定树结构，动态树在各类任务上平均接受率提升 20-40%，尤其在长文本生成场景下优势更大
  • 可配置树大小：支持设置最大树节点数（通常 20-60），在延迟与吞吐之间灵活权衡，适应不同部署场景
  • 端到端加速：在 Vicuna-7B/13B/70B 上实现 3.0-4.0x 加速，在长文本生成场景下加速比更高
  • 兼容 Eagle-v1：Draft Head 架构与 Eagle-v1 完全相同，仅修改了树构建策略，已有 Eagle-v1 的部署可直接升级

面试常问 Q&A

Q1：Eagle-v2 的动态树和 Medusa 的固定树有什么区别？

A：Medusa 使用固定的树拓扑——预先定义好树的形状（例如宽度 5、深度 3），所有分支等概率扩展，不考虑 Draft token 的实际置信度。Eagle-v2 的动态树根据 Draft Head 输出的实际概率来决定每个节点的扩展策略：高概率节点获得更多子节点和更深的扩展，低概率节点提前终止。这使得 Eagle-v2 在相同的节点预算下，期望接受 token 数量显著多于固定树。

Q2：Tree Attention 的 mask 是如何设计的？

A：Tree Attention 的核心是因果 mask 的设计。将树线性化后，每个节点有一个 mask 向量：对于树中的节点 u 和 v，mask[u][v] = 1 当且仅当 v 是 u 的祖先节点或 v 位于 u 的左侧同一层级。这保证了每个节点只能看到其路径上的祖先和已经验证过的同层节点，保持了因果性。实际实现中，mask 矩阵是稀疏的，可以通过 sparse attention 优化计算效率。

Q3：动态树的概率阈值 θ 如何确定？

A：θ 的选择需要在「树的覆盖率」和「树的深度」之间权衡。θ 过高会导致树太窄（只有很少的分支），错过一些可能被接受的 token；θ 过低会导致树太宽太浅（很多低概率分支），浪费计算资源。论文中通过在验证数据集上进行网格搜索来确定最优 θ，通常在 0.1-0.3 之间。实践中也可以根据节点预算动态调整 θ：先用一个初始 θ 构建树，如果节点数未达预算则降低 θ 继续扩展。

Q4：Eagle-v2 的验证开销是否会因为树变大而增加？

A：不会。Tree Attention 的验证开销取决于树的总节点数，而非分支数。Eagle-v2 设定了固定的节点预算（例如 50 个节点），与 Eagle-v1 的固定树节点数相同。因此验证阶段的计算量不变，但由于动态树的接受率更高，每个验证轮次的有效输出 token 更多，所以端到端加速比更大。

Q5：Eagle-v2 相比 EAGLE-1 的训练成本有变化吗？

A：没有。Eagle-v2 的 Draft Head 架构与 Eagle-v1 完全相同，仅修改了推理阶段的树构建策略（从固定树改为动态树）。因此不需要重新训练 Draft Head，已有的 Eagle-v1 预训练权重可以直接用于 Eagle-v2。这是一个重要的工程优势——只需修改推理代码，无需额外训练成本。
