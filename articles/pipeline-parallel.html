    Pipeline Parallelism

    Parallel Strategy — Inter-Layer Distribution
    Pipeline Parallelism（PP）是将 Transformer 的不同层分配到不同 GPU 上的并行策略。例如 GPU 0 处理第 1-10 层，GPU 1 处理第 11-20 层，以此类推。PP 适用于跨节点场景（低带宽、高延迟网络），因为相邻层之间的通信量远小于 TP 的 AllReduce。核心挑战是 Pipeline Bubble——各阶段之间的空闲等待时间。

    核心原理

    当模型太大无法放入单节点时，需要跨多个节点分布。TP 的 AllReduce 通信在跨节点低带宽网络下成为瓶颈，因此跨节点更适合 PP——PP 的通信只发生在相邻阶段之间，通信量为一次激活张量的传输（batch_size × seq_len × hidden_dim），远小于 TP 的 AllReduce。

    基本 PP 调度

    最简单的 PP 调度是 GPipe——将一个 batch 的数据拆分为 M 个 micro-batch，依次注入 Pipeline。GPU 0 先处理 micro-batch 1 的前几层，然后传给 GPU 1，GPU 0 再处理 micro-batch 2 的前几层，以此类推。当所有 micro-batch 都注入后，Pipeline 被填满，各 GPU 可以并行工作。但 GPipe 存在严重的 Pipeline Bubble——在 Pipeline 的填充和排空阶段，部分 GPU 处于空闲状态。Bubble 的比例为 (P-1)/M，其中 P 是 Pipeline 阶段数，M 是 micro-batch 数。当 M >> P 时 Bubble 比例趋近于 0，但在推理场景下（通常 batch 较小），Bubble 比例可能很高。

    1F1B 调度（One Forward One Backward）

    PipeDream 提出的 1F1B 调度优化了 Bubble 时间。在稳态阶段，每个 GPU 交替执行一次 Forward 和一次 Backward，而非先执行所有 Forward 再执行所有 Backward。这减少了中间激活的显存占用（不需要同时保存所有 micro-batch 的激活），同时保持了与 GPipe 相同的 Bubble 比例。1F1B 是 Megatron-LM 和 DeepSpeed 的默认 PP 调度策略。

    Zero Bubble Pipeline Parallelism（2024）

    Zero Bubble PP（Qi et al., 2024）是 PP 调度的最新突破。核心思想是将 Backward 拆分为两个独立的操作：B（计算输入梯度，用于传给前一个阶段）和 W（计算权重梯度，用于参数更新）。通过重新调度 B 和 W 的执行顺序，使得 Pipeline 中没有任何 GPU 处于空闲状态——理论上实现了零 Bubble。在训练场景下，Zero Bubble PP 比 1F1B 提升 10-30% 的训练吞吐量。在推理场景下，由于没有 Backward，Zero Bubble 的思想可以简化为「微批次流水线」——将 Prefill 拆分为多个 micro-batch 流水线执行。

    PP 在推理中的应用

    在推理场景下，PP 的应用与训练略有不同：（1）没有 Backward，只有 Forward，Bubble 的影响更大；（2）batch size 通常较小（推理的 batch 受延迟约束），micro-batch 数量有限；（3）Prefill 阶段可以使用较大的 micro-batch 来填充 Pipeline，Decode 阶段由于每步只生成一个 token，Pipeline 效率较低。因此推理系统通常将 TP 和 PP 结合使用——节点内用 TP（高带宽），节点间用 PP（低带宽）。例如 LLaMA-2-70B 在 2 节点 16 GPU 上的配置：TP=8（节点内）× PP=2（跨节点）。

    关键特性

    • 层间切分：将 Transformer 的不同层分配到不同 GPU，每个 GPU 处理连续的若干层
    • 适用于跨节点：通信只在相邻阶段之间，通信量远小于 TP，适合低带宽跨节点场景
    • 显存效率：每个 GPU 只需存储其负责层的权重和激活，显存占用按阶段数均分
    • Pipeline Bubble：核心挑战——填充和排空阶段部分 GPU 空闲，Bubble 比例为 (P-1)/M
    • Zero Bubble（2024）：通过拆分 Backward 操作和重新调度，理论上实现零 Bubble
    • 与 TP 组合：实践中 TP+PP 组合使用——节点内 TP，节点间 PP

    面试常问 Q&A

    Q1：Pipeline Parallelism 和 Tensor Parallelism 的区别是什么？各自适用于什么场景？

    A：TP 是层内切分（将单个层的权重切分到多个 GPU），PP 是层间切分（将不同层分配到不同 GPU）。TP 的通信频率高（每层 2 次 AllReduce），需要高带宽互联，适合节点内。PP 的通信频率低（只在相邻阶段之间传输激活），适合跨节点低带宽场景。实践中通常 TP+PP 组合：节点内 TP=8（利用 NVLink），节点间 PP=2-4（利用 InfiniBand）。

    Q2：什么是 Pipeline Bubble？如何减少？

    A：Pipeline Bubble 是 Pipeline 填充和排空阶段的 GPU 空闲时间。当第一个 micro-batch 到达最后一个阶段时，前面的阶段还在等待后续 micro-batch，产生空闲。Bubble 比例为 (P-1)/M（P=阶段数，M=micro-batch 数）。减少方法：（1）增加 micro-batch 数量 M——M 越大 Bubble 比例越小，但受 batch size 限制；（2）使用 1F1B 调度——交替 Forward/Backward，减少显存占用；（3）Zero Bubble PP——拆分 Backward 操作，理论上消除 Bubble。

    Q3：PP 在推理场景下的效率如何？

    A：PP 在推理场景下的效率低于训练场景，原因：（1）没有 Backward，只有 Forward，Bubble 无法被 Backward 填充；（2）推理的 batch size 较小（受延迟约束），micro-batch 数量有限，Bubble 比例更高；（3）Decode 阶段每步只生成一个 token，Pipeline 的并行度很低。因此推理系统通常只在跨节点时使用 PP，且尽量减少 PP 的阶段数（通常 PP=2-4）。节点内优先使用 TP。

    Q4：Zero Bubble PP 是如何实现零 Bubble 的？

    A：核心思想是将 Backward 拆分为两个操作：B（计算输入梯度，传给前一阶段）和 W（计算权重梯度，更新参数）。传统 1F1B 中，Backward 是一个整体，执行时前一阶段必须等待。Zero Bubble 将 B 和 W 解耦——B 的执行不依赖 W，可以更早地触发前一阶段的计算。通过精心调度 B 和 W 的执行顺序，使得每个 GPU 在每个时间步都有计算任务，没有空闲。论文证明了在理想条件下（计算时间均匀），可以实现零 Bubble。

    Q5：跨节点推理的最优并行配置是什么？

    A：以 LLaMA-2-70B 在 2 节点 16 GPU（8 GPU/节点，NVLink 节点内，InfiniBand 跨节点）为例：最优配置通常是 TP=8 × PP=2。理由：（1）节点内 TP=8 利用 NVLink 高带宽，AllReduce 延迟极低；（2）跨节点 PP=2，通信量仅为一次激活传输，InfiniBand 可以承受；（3）如果使用 TP=16（跨节点），AllReduce 在 InfiniBand 上的延迟会成为严重瓶颈。如果模型更大（如 405B），可能需要 TP=8 × PP=4 或 TP=8 × PP=8。
