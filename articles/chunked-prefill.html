    Chunked Prefill

    Inference Framework — Prefill-Decode Interleaving
    Chunked Prefill 是一种将长 Prompt 的 Prefill 计算拆分为多个小块（Chunk），与 Decode 步骤交错执行的调度策略。它解决了长 Prompt Prefill 阻塞 Decode 的问题，显著降低了 Decode 请求的延迟（TTBT），同时保持了 GPU 的高利用率。vLLM v0.5+ 和 SGLang 均已原生支持 Chunked Prefill。

    核心原理

    在 Continuous Batching 中，新请求需要先执行 Prefill 才能加入 Decode 批次。如果新请求的 Prompt 很长（例如 4K-8K token），Prefill 需要数十毫秒甚至数秒，在此期间所有正在 Decode 的请求都被阻塞——这就是「Prefill Stall」问题。Prefill Stall 会导致 Decode 请求的 Token-to-Token Time（TTBT）急剧增加，用户体验恶化。

    Prefill Stall 问题详解

    Prefill 是 compute-bound——需要对整个 Prompt 进行一次完整的前向传播，计算量与 Prompt 长度成正比。Decode 是 memory-bound——每步只需计算一个 token 的前向传播，但需要读取整个 KV Cache。当一个长 Prompt 的 Prefill 与 Decode 混在同一轮时，Prefill 的长计算时间会阻塞 Decode：所有 Decode 请求必须等待 Prefill 完成才能生成下一个 token。假设 Prefill 耗时 100ms，Decode 批次中有 32 个请求，那么这 32 个请求的 TTBT 都会增加 100ms——这就是 Prefill Stall。

    Chunked Prefill 的拆分策略

    Chunked Prefill 的核心思想是将长 Prefill 拆分为多个小块（Chunk），每个 Chunk 的大小通常与 Decode 批次的 token 数量相当（例如 512-2048 token）。拆分后，每一轮 Forward Pass 中同时包含：（1）一个 Prefill Chunk（处理 Prompt 的一小部分）；（2）整个 Decode 批次的所有 Decode token。这样 Prefill 的计算量被分散到多轮中，每轮的计算时间不会显著增加，Decode 请求的 TTBT 保持稳定。

    具体流程：假设一个新请求的 Prompt 长度为 4096 token，Chunk 大小为 1024。拆分为 4 个 Chunk：（1）第 1 轮：执行 Chunk-1（token 0-1023）的 Prefill + 所有 Decode 请求的 Decode；（2）第 2 轮：执行 Chunk-2（token 1024-2047）的 Prefill + 所有 Decode 请求的 Decode；（3）第 3-4 轮：类似处理 Chunk-3 和 Chunk-4。4 轮完成后，新请求的 Prefill 全部完成，加入 Decode 批次。整个过程中 Decode 请求的 TTBT 增量仅为一个 Chunk 的 Prefill 时间（约 25ms），而非整个 Prompt 的 Prefill 时间（约 100ms）。

    Chunk 大小的选择

    Chunk 大小的选择需要在 TTBT 延迟和整体效率之间权衡：（1）Chunk 越小，每轮 Prefill 的计算量越小，Decode TTBT 增量越小，但 Prefill 的总轮数更多，调度开销更大；（2）Chunk 越大，Prefill 轮数更少，调度开销更小，但每轮 Prefill 的计算量更大，Decode TTBT 增量更大。实践中通常选择 512-2048 token 作为 Chunk 大小，这是一个在延迟和效率之间的平衡点。vLLM 默认 Chunk 大小为 2048，SGLang 默认为 512。

    与 Continuous Batching 的协同

    Chunked Prefill 是 Continuous Batching 的自然扩展。在调度层面，每一轮的 Forward Pass 由以下部分组成：（1）0-N 个 Prefill Chunk（来自新请求的不同 Chunk）；（2）所有活跃 Decode 请求的 Decode token。调度器需要决定每一轮放入多少 Prefill Chunk——过多会增加 Decode TTBT，过少会浪费 GPU 计算能力。一个简单的启发式规则是：每轮 Prefill Chunk 的总 token 数不超过 Decode 批次总 token 数的 2 倍。

    Prefill-Decode 分离架构

    Chunked Prefill 的进一步演进是 Prefill-Decode 分离架构（Disaggregated Serving）。在这种架构中，Prefill 和 Decode 运行在不同的 GPU/节点上：Prefill 节点专门处理 Prompt 的 Prefill 计算，Decode 节点专门处理自回归生成。Prefill 完成后，将 KV Cache 传输到 Decode 节点。这种分离使得 Prefill 和 Decode 可以独立扩缩容——当 Prefill 负载高时增加 Prefill 节点，当 Decode 负载高时增加 Decode 节点。代表系统：DistServe（2024）、Splitwise（2024）、Mooncake（2024）。

    关键特性

    • Prefill 拆分：将长 Prompt 的 Prefill 拆分为多个小块（Chunk），与 Decode 交错执行，消除 Prefill Stall
    • TTBT 降低 3-5x：Decode 请求的 Token-to-Token Time 增量从整个 Prefill 时间降低为单个 Chunk 时间
    • GPU 利用率保持高位：Prefill Chunk 与 Decode 混合执行，GPU 始终有计算任务，利用率接近 100%
    • 可配置 Chunk 大小：支持 512-2048 token 的 Chunk 大小，在延迟和效率之间灵活权衡
    • 支持 Prefill-Decode 分离：可扩展为 Prefill/Decode 运行在不同 GPU 的分离架构，独立扩缩容
    • 行业支持：vLLM v0.5+、SGLang、TensorRT-LLM 均已原生支持

    面试常问 Q&A

    Q1：Chunked Prefill 解决了什么问题？

    A：解决了长 Prompt 的 Prefill 阻塞 Decode 的问题（Prefill Stall）。当一个长 Prompt（如 4K-8K token）进入系统时，其 Prefill 计算需要数十毫秒到数秒，在此期间所有正在 Decode 的请求都被阻塞，TTBT 急剧增加。Chunked Prefill 将 Prefill 拆分为多个小块，与 Decode 交错执行，使得 Decode 请求的 TTBT 增量仅为一个 Chunk 的 Prefill 时间。

    Q2：Chunked Prefill 的 Chunk 大小如何选择？

    A：Chunk 大小通常为 512-2048 token。选择依据：（1）延迟需求——如果对 TTBT 延迟敏感，选择较小的 Chunk（512），每轮 Prefill 时间更短；（2）吞吐需求——如果更关注吞吐量，选择较大的 Chunk（2048），减少调度开销；（3）GPU 规模——大 GPU（如 H100）的计算能力更强，可以承受更大的 Chunk。vLLM 默认 2048，SGLang 默认 512。实践中通常在目标负载下进行 benchmark 来确定最优值。

    Q3：Chunked Prefill 和 Prefix Caching 可以同时使用吗？

    A：可以，且效果叠加。Prefix Caching 缓存共享前缀的 KV Cache，减少需要 Prefill 的 token 数量；Chunked Prefill 将剩余需要 Prefill 的 token 拆分为小块。两者结合：首先通过 Prefix Caching 跳过已缓存的前缀部分，然后对未缓存的部分使用 Chunked Prefill 逐步处理。在系统提示词共享的场景下，Prefix Caching 可以跳过大部分 Prefill，Chunked Prefill 只需处理少量的用户输入部分。

    Q4：Prefill-Decode 分离架构的优势是什么？

    A：分离架构将 Prefill 和 Decode 运行在不同的 GPU 上，优势在于：（1）独立扩缩容——Prefill 是 compute-bound，需要高算力 GPU；Decode 是 memory-bound，需要大显存。分离后可以针对各自特点选择最优硬件；（2）互不干扰——Prefill 的长计算不会阻塞 Decode，Decode 的 KV Cache 读取不会与 Prefill 的计算竞争 GPU 资源；（3）资源利用率更高——可以根据实时负载动态调整 Prefill/Decode 的 GPU 比例。劣势是需要在 GPU 之间传输 KV Cache，网络带宽成为瓶颈。

    Q5：Chunked Prefill 对 Prefill 总时间有影响吗？

    A：有轻微影响。Chunked Prefill 将 Prefill 拆分为多个 Chunk 与 Decode 交错执行，每轮 Prefill Chunk 的计算会与 Decode 的计算共享 GPU 资源，因此 Prefill 的总时间会略有增加（约 10-20%）。但这个代价是值得的——换来的是 Decode 请求的 TTBT 大幅降低。在高并发场景下，如果没有 Chunked Prefill，长 Prompt 的 Prefill 会导致所有 Decode 请求的 TTBT 飙升，用户体验急剧恶化。Chunked Prefill 通过牺牲少量 Prefill 效率，换取了整体系统的延迟稳定性。
